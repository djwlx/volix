# 随机图片缓存目录首张随机设计

## 背景

当前 `随机图片配置` 页面可以看到缓存目录列表，但用户无法从某个缓存目录直接进入随机图片浏览。现有 `随机图片` 页面虽然已经支持基于当前图片继续做 `同文件夹随机`，但缺少“从指定缓存目录开始看第一张图”的入口。

本次改动目标是：

- 在 `随机图片配置` 页的缓存目录列表中新增 `随机` 操作。
- 点击后跳转到 `随机图片` 页面。
- 首次进入时从该缓存目录内随机取一张图片。
- 进入页面后保持现有两个按钮语义不变：
  - `换一张` 继续全局随机。
  - `同文件夹随机` 继续按当前图片所在文件夹随机。

## 用户体验

### 配置页

在缓存目录列表每一行的操作列新增一个 `随机` 按钮，仅对可随机的缓存目录显示可用状态。

- 点击后跳转到随机图片页面。
- 跳转时把当前缓存目录的 `cid` 作为查询参数带过去。

### 随机图片页

当页面 URL 中带有缓存目录 `cid` 参数时：

- 首次加载不走全局随机。
- 首次改为调用新的“按缓存目录随机”接口。
- 拿到首张图后，页面后续操作沿用当前逻辑。

当页面没有该参数时：

- 页面行为与现在完全一致。

## 后端设计

### 新增接口

新增一个按缓存目录随机首张图片的接口：

- 路由：`GET /115/pic/cache-random`
- 参数：`cid`
- 返回：沿用 `Random115PicResponse`

### 服务逻辑

接口核心流程如下：

1. 校验 `cid` 非空。
2. 以该 `cid` 作为候选缓存目录范围，从本地文件缓存表中随机取一张图片。
3. 复用现有随机图片元信息拼装逻辑，返回 `url`、`fileName`、`pc`、`cid`、`path`、`parentPath`、`liked` 等字段。
4. 继续沿用现有去重窗口逻辑，尽量避免最近取过的图片重复出现。

### 约束

- 本次只支持对已经进入随机缓存体系的目录做随机，不扩展到任意 115 目录实时取图。
- 查不到候选图片时返回明确错误。
- 返回模型保持与现有随机图片接口一致，避免前端增加额外分支。

## 前端设计

### 配置页列表操作

在缓存目录表格操作列新增：

- `随机`：跳到 `/pic?cid=<当前目录cid>`

不新增说明文案，不改变已有删除、重试行为。

### 随机图片页首屏分流

页面首次加载时：

- 如果存在 `cid` 查询参数，调用新接口获取首张图。
- 否则调用现有全局随机接口。

页面内按钮行为保持不变：

- `换一张` 始终调用全局随机接口。
- `同文件夹随机` 始终调用现有父目录随机接口。

## 测试范围

### 后端

- 新接口在传入合法 `cid` 时返回随机图片元信息。
- 新接口会把 `cid` 传递到随机查询层。
- `cid` 为空时返回参数错误。

### 前端

- 缓存目录列表会渲染新的 `随机` 操作。
- 点击后会跳转到带 `cid` 查询参数的随机图片页面。
- 随机图片页在有 `cid` 时首屏调用新接口。
- 页面后续点击 `换一张` 仍走全局随机。

## 验收标准

- 配置页缓存目录列表新增 `随机` 操作。
- 从该操作进入随机图片页后，首张图片来自所选缓存目录。
- 页面现有 `换一张` 和 `同文件夹随机` 行为不回退。
- 无新增硬编码文案，新增文案全部进入 i18n。
